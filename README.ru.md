# TON Testnet Self-Custodial Wallet (Frontend MVP)

[English version](./README.md)

Некастодиальный кошелек для **тестовой сети TON**, работающий целиком в браузере.
Собственного серверного API нет. Пользовательские данные хранятся локально.

## Что реализовано

- Онбординг:
  - Создание нового кошелька в тестовой сети.
  - Показ seed-фразы и обязательное подтверждение перед продолжением.
  - Импорт существующего кошелька по seed-фразе.
- Локальное управление несколькими кошельками:
  - Добавление нескольких кошельков.
  - Переключение активного кошелька в верхней панели.
  - Автопереход на Home после переключения.
- Home:
  - Адрес кошелька.
  - Текущий баланс в TON.
  - Последние транзакции.
  - Поиск по транзакциям (адрес, хэш, сумма, тип).
- Receive:
  - Показ адреса кошелька.
  - Копирование адреса с всплывающим уведомлением.
- Send:
  - Проверка адреса и суммы.
  - Блок подтверждения перед отправкой.
  - Статусы: отправка, ожидание подтверждения, ошибка.
  - Замена локального `pending` на реальный хэш после подтверждения.
- Обновления в реальном времени:
  - Подписка на транзакции через WebSocket (TON Center streaming API).
  - Keepalive ping.
  - Резервный периодический опрос при `disconnected/error/reconnecting`.
- Состояния интерфейса:
  - `loading/success/error` для обновления данных и отправки.
  - Всплывающие уведомления через SweetAlert2 (включая загрузку).
- Мобильная адаптация:
  - Адаптивные верхняя панель, навигация, карточки, кнопки и список транзакций.

## Технологический стек

- React + Vite + TypeScript

## Переменные окружения

Скопируйте `.env.example` в `.env` и при необходимости измените значения.

| Переменная | Значение по умолчанию | Описание |
|---|---|---|
| `VITE_TON_TESTNET_ENDPOINT` | `https://testnet.toncenter.com/api/v2/jsonRPC` | Основной RPC-адрес тестовой сети TON |
| `VITE_TONCENTER_API_KEY` | пусто | Единый ключ TON Center для RPC и WebSocket |
| `VITE_TON_TESTNET_FALLBACK_ENDPOINTS` | пусто | Запасные RPC-адреса через запятую |
| `VITE_TON_TESTNET_WS_ENDPOINT` | `wss://testnet.toncenter.com/api/streaming/v2/ws` | WebSocket-адрес для потоковых событий |
| `VITE_TONCENTER_STREAM_TOKEN_PARAM` | `api_key` | Имя query-параметра для ключа (`token` или `api_key`) |
| `VITE_TONCENTER_STREAM_MIN_FINALITY` | `pending` | Минимальная финальность событий |
| `VITE_ENABLE_WS_STREAM` | `true` | Включение/выключение WebSocket-стрима |
| `VITE_TON_EXPLORER_TX_BASE` | `https://testnet.tonviewer.com/transaction` | Базовый URL обозревателя транзакций |

### Рабочий пример `.env`

```env
VITE_TON_TESTNET_ENDPOINT=https://testnet.toncenter.com/api/v2/jsonRPC
VITE_TONCENTER_API_KEY=<ВАШ_КЛЮЧ_TONCENTER>
VITE_TON_EXPLORER_TX_BASE=https://testnet.tonviewer.com/transaction
VITE_ENABLE_WS_STREAM=true
VITE_TONCENTER_STREAM_TOKEN_PARAM=api_key
```

Примечания:
- Если ваш ключ для стрима должен передаваться как `token`, используйте `VITE_TONCENTER_STREAM_TOKEN_PARAM=token`.
- Для отказоустойчивости можно дополнительно задать `VITE_TON_TESTNET_FALLBACK_ENDPOINTS`.

### Где взять ключи

1. Откройте сайт TON Center: `https://toncenter.com/`.
2. Войдите в аккаунт и создайте API-ключ в разделе API.
3. Укажите этот ключ в `VITE_TONCENTER_API_KEY`.

Обратная совместимость:
- `VITE_TON_TESTNET_API_KEY` и `VITE_TONCENTER_STREAM_TOKEN` всё ещё поддерживаются как запасной вариант, но предпочтительно использовать только `VITE_TONCENTER_API_KEY`.

Важно по безопасности:
- Не добавляйте реальные ключи в git.
- Если ключ уже был опубликован, отзовите и перевыпустите его в TON Center.

## Архитектура и компромиссы

Подробно: [docs/ARCHITECTURE.md](./docs/ARCHITECTURE.md)

### Почему такая архитектура

1. FSD-слои разделяют маршруты, доменную логику и общие компоненты.
2. Логика кошелька и транзакций изолирована в `entities/*`, поэтому изменения интерфейса не ломают сетевую часть.
3. Ключевые пользовательские действия (`send/receive/search`) вынесены в `features/*`, что упрощает тестирование.
4. Хранилище и TON-адаптеры находятся рядом с доменом (`entities/wallet`), интеграция с сетью централизована.
5. Отказ от собственного backend соответствует условиям задания и снижает сложность запуска.

### Почему выбран этот стек

- React + TypeScript: предсказуемая модель компонентов и строгая типизация.
- Vite: быстрый цикл разработки и простая сборка.
- Zustand (+ persist): минимум шаблонного кода, локальная персистентность без лишней сложности.
- `@ton/ton` + `@ton/crypto`: нативные библиотеки TON для адресов, транзакций и отправки.
- TON Center RPC + Streaming WS: чтение ончейн-данных и обновления в реальном времени без backend.
- Vitest: легковесные модульные тесты в том же TS/Vite-контуре.

### Компромиссы и почему они приняты

- Хранение seed/mnemonic в localStorage.
Причина: frontend-only MVP и отсутствие backend.
Риск: небезопасно при компрометации устройства.
План: шифрование с passphrase (P0).

- Приближенная комиссия (`ESTIMATED_FEE_TON`) без симуляции.
Причина: быстрее реализация и проще UX.
Риск: оценка комиссии не точная.
План: добавить более точную оценку/симуляцию.

- Оптимистичная запись `pending` до финального хэша.
Причина: в testnet бывают задержки индексации, нужен мгновенный отклик.
Риск: временное расхождение локального статуса и данных сети.
Снижение риска: сверка и повторы до замены `pending` на подтвержденный хэш.

- WebSocket + резервный опрос только при деградации WebSocket.
Причина: баланс между отзывчивостью и нагрузкой.
Риск: зависимость от доступности внешнего стрима.
Снижение риска: reconnect, keepalive и fallback polling.

- Только клиентские проверки защиты от подмены адреса.
Причина: no-backend ограничение.
Риск: эвристики слабее полноценной антифрод-системы.
План: усиление алгоритмов похожести и модель доверенных контактов.

## Безопасность интерфейса / Защита от подмены адреса

Реализовано в сценарии отправки:

1. Каноническая нормализация и сравнение адресов.
- Адрес приводится к canonical-форме (raw/bounceable/non-bounceable/url-safe).
- Сравнение выполняется по raw.

2. Предупреждение для нового адреса.
- Если получатель не встречался в локальной истории/списке известных адресов, показывается warning.

3. Предупреждение для похожего адреса.
- Адрес считается рискованным, если совпадают начало и конец, но отличается середина.

4. Явное подтверждение рискованной отправки.
- Пока пользователь не поставит чекбокс, отправка блокируется.

5. Дополнительное подтверждение для крупных сумм.
- Для суммы `>= 20 TON` нужно ввести последние 4 символа адреса получателя.

1. `src/entities/wallet/lib/address.test.ts`
- Проверка `normalizeTonAddress`.
- Проверка канонического сравнения `areSameTonAddress`.

2. `src/features/send-ton/model/validation.test.ts`
- Обязательность полей.
- Отклонение неверного адреса.
- Проверка `amount + fee <= balance`.
- Возврат нормализованного адреса при валидных данных.

3. `src/features/send-ton/model/risk-checks.test.ts`
- Новый адрес помечается как рискованный.
- Известный адрес с небольшой суммой не помечается как рискованный.
- Похожий адрес + крупная сумма требуют дополнительных подтверждений.

4. `src/entities/transaction/lib/transaction-mapper.test.ts`
- Корректное извлечение hash, когда hash приходит как функция.
- Корректный маппинг ext-in signed transaction с исходящим value как исходящей транзакции.

## Ограничения (MVP)

- Только тестовая сеть TON.
- Нет backend и синхронизации между устройствами.
- Хранение seed/mnemonic в localStorage не подходит для production.
- Нет интеграции с аппаратными кошельками.
- Нет точной симуляции комиссии.

## Дальнейшие улучшения (приоритет)

1. P0: Шифрование seed/mnemonic в хранилище браузера (WebCrypto).
2. P0: Улучшить надежность отслеживания подтверждений и резолва хэша.
3. P1: Доверенная адресная книга (метки, whitelist) и более сильные anti-spoof проверки.
4. P1: QR для получения и отправки.
5. P2: E2E smoke-тесты (Playwright) для onboarding/send/receive.
6. P2: Поддержка Jetton и расширенные фильтры/пагинация истории.
